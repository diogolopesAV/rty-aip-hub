# Riverty AI Package Hub — Proposal

The existing Skillshare-based skills distribution does not support hooks, MCP servers, or plugins. APM fills that gap.

APM (Agent Package Manager) is Microsoft's open-source package manager for AI agent primitives. It fetches packages from GitHub and deploys their contents into the right directories for whichever AI tools a developer is using — Claude Code, Cursor, Copilot, or Codex — from a single install command.

```bash
curl -sSL https://aka.ms/apm-unix | sh
```

---

## What APM Supports

### Primitive Types

| Primitive       | What it is                                                                    |
| --------------- | ----------------------------------------------------------------------------- |
| **Skill**       | A scoped capability with instructions, scripts, and reference material        |
| **MCP Server**  | A tool-server declaration auto-wired into the developer's AI tools            |
| **Hook**        | An event handler triggered by runtime events (e.g. pre-commit, session start) |
| **Command**     | A slash-command shortcut or bundled collection of primitives                  |
| **Instruction** | A long-lived behavioral rule applied across all sessions                      |
| **Agent**       | A named persona with explicit scope, tools, and triggers                      |

### Package Types (Auto-Detected)

APM infers the package type from the source repo structure — no configuration needed in the source repo:

| Package Type | Detection signal | What it installs |
|---|---|---|
| **Skill Bundle** | `SKILL.md` at root | One skill (atomic unit) |
| **Skill Collection** | `skills/<name>/SKILL.md` | Multiple skills, installable individually |
| **Plugin Collection** | `plugin.json` at root | Skills + commands + hooks + MCPs as declared |
| **APM Package** | `.apm/` directory | Mixed primitives: agents, instructions, prompts, skills |

### Where Primitives Deploy

| Primitive | Claude Code | Cursor | Copilot |
|---|---|---|---|
| Skills | `.claude/skills/` | `.cursor/rules/` | `.github/copilot-instructions.md` |
| Instructions | `.claude/rules/` | `.cursor/rules/` | `.github/instructions/` |
| Commands | `.claude/commands/` | — | — |
| Hooks | `.claude/settings.json` | — | — |
| MCP Servers | `.mcp.json` | `.cursor/mcp.json` | — |
| Agents | `.claude/agents/` | — | — |

## Hub — `rty-aip-hub`

> **MVP**: [github.com/diogolopesAV/rty-aip-hub](https://github.com/diogolopesAV/rty-aip-hub) — a proof-of-concept. Production implementation would live under the Riverty organization.

The existing `skills` repository hosts skill files that teams use for AI context. `rty-aip-hub` is that same repository re-implemented using APM conventions — same content, but now discoverable through a marketplace, installable with a single command, and deployable to any supported AI tool with no manual file-copying. It serves two roles: a **marketplace** (register once, install by name) and a **content host** (Platform Team-authored packages live directly in the repo).

```
rty-aip-hub/
├── apm.yml                         ← marketplace catalog (marketplace.name = "rty-hub")
├── .claude-plugin/
│   └── marketplace.json            ← generated by CI; consumed by apm marketplace add
├── plugins/                       ← hub-authored packages (equivalent to skill folders in the skills repo)
│   ├── rty-atlassian-mcp/
│   ├── rty-skill-authoring/
│   ├── rty-web-accessibility/
│   └── rty-terraform-azure-cdn-classic-migration/
└── bundles/
    ├── ai/apm.yml
    └── bip/apm.yml
```

### plugins/

The direct equivalent of the skill folders in the existing `skills` repository — but now each is an APM-managed package. Unlike the old flat skill files, each subdirectory can contain any combination of APM primitives: skills, MCPs, hooks, commands, agents. Platform Team owns and maintains these. External repos — like the BIP team skills, which live in their own private GitHub repos — are referenced in `apm.yml` by source rather than copied in. The `apm.yml` at the hub root is the single source of truth listing all packages, both hub-authored and external.

### bundles/

The equivalent of a curated team list. Instead of browsing the full catalog and installing packages one by one, a developer installs a bundle and gets everything their team needs in one shot:

- `bundles/ai/` — Atlassian MCP setup + skill authoring guidelines
- `bundles/bip/` — BIP Python + Terraform coding standards

A bundle is a lightweight `apm.yml` with a dependencies list — no extra tooling, no separate pipeline.

### Central `apm.yml`

The `apm.yml` at the hub root is the single source of truth. It declares the marketplace catalog and the full set of installable packages in one file:

```yaml
name: rty-aip-hub
version: 1.0.0
description: "Riverty AI Package Hub — curated skills and AI capabilities for the Riverty platform"

targets:
  - claude
  - cursor
  - codex
  - copilot

marketplace:
  name: rty-hub
  owner:
    name: Riverty Technology Innovation
    url: https://github.com/Riverty-Tech-Innovation

  packages:
    # Hub-authored — content lives in plugins/ in this repo
    - name: rty-atlassian-mcp
      source: ./plugins/rty-atlassian-mcp
      description: "Configure Atlassian's cloud MCP server for Jira and Confluence access in AI agents"
      tags: [skill, ai, confluence, jira, mcp]

    - name: rty-skill-authoring
      source: ./plugins/rty-skill-authoring
      description: "Create, review, and refine agent skills following Riverty conventions"
      tags: [skill, ai, workflow]

    - name: rty-web-accessibility
      source: ./plugins/rty-web-accessibility
      description: "Ensure generated HTML/CSS is WCAG 2.1/2.2 Level AA compliant using Riverty web components"
      tags: [skill, design-system, accessibility, html, css]

    - name: rty-terraform-azure-cdn-classic-migration
      source: ./plugins/rty-terraform-azure-cdn-classic-migration
      description: "Guide Azure CDN Classic to Azure Front Door migration for Terraform-managed environments"
      tags: [skill, azure, terraform, migration]

    # External — private Riverty repos, SHA-pinned (branch refs are rejected by apm pack)
    - name: bip-python-skill
      source: Riverty-Tech-Innovation/bip-python-skill
      ref: a54a2102db113824db71cccf1241cdae62d2ebdb
      description: "Python coding practices following BIP standards"
      tags: [skill, bip, python]

    - name: bip-terraform-skill
      source: Riverty-Tech-Innovation/bip-terraform-skill
      ref: 89fe35c95bcf9adffa1a1d03118f9cfa7285924f
      description: "Terraform IaC coding practices following BIP standards"
      tags: [skill, bip, terraform, iac]

# Full baseline: apm install diogolopesAV/rty-aip-hub installs all 6 packages.
# Hub-authored packages use remote paths (not ./plugins/)
dependencies:
  apm:
    - diogolopesAV/rty-aip-hub/plugins/rty-atlassian-mcp
    - diogolopesAV/rty-aip-hub/plugins/rty-skill-authoring
    - diogolopesAV/rty-aip-hub/plugins/rty-web-accessibility
    - diogolopesAV/rty-aip-hub/plugins/rty-terraform-azure-cdn-classic-migration
    - Riverty-Tech-Innovation/bip-python-skill#a54a2102db113824db71cccf1241cdae62d2ebdb
    - Riverty-Tech-Innovation/bip-terraform-skill#89fe35c95bcf9adffa1a1d03118f9cfa7285924f
```

Two things worth noting here. External packages (BIP) are pinned to a full commit SHA rather than a branch — APM rejects branch refs as mutable at pack time. Hub-authored packages reference themselves via their remote GitHub path, not `./plugins/`, because local `path:` refs are also blocked; APM fetches them from GitHub during install.

### Registering

```bash
# One-time per machine — fetches the marketplace manifest from the hub
apm marketplace add diogolopesAV/rty-aip-hub
```

```
Registered Marketplaces
┌─────────┬──────────────────────────┬──────┬─────────────────────────────────┐
│ Name    │ Source                   │ Ref  │ Path                            │
├─────────┼──────────────────────────┼──────┼─────────────────────────────────┤
│ rty-hub │ diogolopesAV/rty-aip-hub │ main │ .claude-plugin/marketplace.json │
└─────────┴──────────────────────────┴──────┴─────────────────────────────────┘
```

The catalog behind this registration is always current. Every PR that touches `apm.yml` or `plugins/**` runs `apm pack --dry-run --check-clean` to validate source refs. On merge, CI runs `apm pack` and commits the updated `.claude-plugin/marketplace.json` back to main automatically — no manual step required.

### Installing

Global (`-g`) deploys to `~/.claude/skills/` — available in all Claude sessions, no files added to any project repo. Without `-g`, APM installs into the current directory and creates `apm.yml` and `apm.lock.yaml`.

```bash
# Single package globally — available in all Claude sessions, no files added to your project
apm install -g rty-web-accessibility@rty-hub
```

## Key Commands

### Setup

```bash
curl -sSL https://aka.ms/apm-unix | sh   # install CLI
apm config set target claude             # set default target so --target isn't needed every time
apm config list                          # show current config (target, auto-integrate, etc.)
```

### Marketplace

```bash
apm marketplace add owner/repo           # register a hub (auto-discovers marketplace.json)
apm marketplace browse marketplace-name  # list available packages
apm marketplace list                     # show all registered marketplaces on this machine
apm marketplace remove marketplace-name  # unregister a marketplace
apm marketplace update                   # refresh cached marketplace manifests
```

### Install

```bash
# By catalog name (requires marketplace add first)
apm install name@marketplace-name
apm install -g name@marketplace-name          # global: deploys to ~/.claude/skills/, no project files created

# By direct path (no marketplace registration needed)
apm install owner/repo --target claude
apm install owner/repo#sha --target claude    # pin to a specific commit SHA or tag
apm install owner/repo/bundles/team --target claude

# Selective install
apm install name@hub --skill skill-name       # cherry-pick one skill from a collection
apm install owner/repo --only apm             # skills only, skip MCPs
apm install owner/repo --only mcp             # MCPs only, skip skills

# Flags
apm install owner/repo --trust-transitive-mcp # required when a hub dependency includes an MCP server*
apm install owner/repo --dry-run              # preview what would be installed without doing it
apm install owner/repo --refresh              # re-fetch all deps from upstream (bypass cache)
apm install owner/repo --target claude,cursor # deploy to multiple harnesses at once
```

> **Auth**: APM uses `gh` CLI credentials automatically when `gh auth login` is active. For private repos without `gh`: `export GITHUB_APM_PAT=ghp_your_token`.

> *Installing a package directly auto-wires its MCPs. When the package is installed transitively through a hub, APM blocks MCPs by default — `--trust-transitive-mcp` grants explicit consent.

### Manage

```bash
apm uninstall -g name@marketplace-name   # uninstall by same identifier used to install
apm uninstall -g owner/repo/path

apm update                               # update all deps to their latest matching refs
apm outdated                             # show which installed packages have upstream updates
apm deps list                            # installed packages in the current project
apm deps list -g                         # globally installed packages
apm prune                                # remove packages no longer listed in apm.yml
```

## Contribution

Contributions flow through a frontend UI, both paths resulting in a GitHub PR reviewed by the Platform Team.

**External submission**: contributor references an existing GitHub repo and specifies the type (skill, plugin, MCP). Content stays in the source repo; the hub receives a catalog entry pointing to it. The source repo needs no APM configuration of its own.

**Internal submission**: contributor authors content directly in the UI — skill body, assets, and references. The frontend creates the files under `plugins/` in the hub repo.
